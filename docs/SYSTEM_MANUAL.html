2. Security & Compliance

Security is not an afterthought but a core architectural pillar. The system implements a Zero-Trust model where every request is verified.

2.1 Authentication Flow

We use JSON Web Tokens (JWT) for stateless authentication. This allows the API to scale infinitely without server-side session storage bottlenecks.

Security Feature: IP Rate Limiting To prevent brute-force attacks, the firewall monitors login attempts. If an IP fails 5 times within 1 minute, it is automatically blocked for 60 minutes. This happens at the infrastructure layer, protecting the database from load.

2.2 Role-Based Access Control (RBAC)

The system enforces strict permission scopes:

  • Company Admin: Full CRUD access to all tables. Can "Lock" schools.
  • School User: Read/Write to their own students only. When is_locked=1, they are downgraded to Read-Only.
  • Production Staff: Write access only to status fields. Cannot delete data.

3. High-Performance Features

3.1 Auto-Heal Engine ("The Brain")

A distinctive feature of this system is the Auto-Heal Engine, a proprietary self-repairing reliability layer designed and built by Anson Saju.

Architected by Anson Saju The "Brain" runs on every server restart. It proactively:
  • Verifies Schema Integrity: Checks 50+ columns for data type mismatches.
  • Re-Seeds Configuration: If critical rows (like Production Stages) are deleted, it regenerates them from an immutable code backup within 50ms.
  • Prevents Downtime: This design ensures zero downtime even during database glitches.

3.2 Global Deployment & Data Freedom

  • Containerization: The app runs in Docker containers, making it agnostic to the host OS.
  • Full Data Dump: Admins can export the entire database state to JSON/SQL at any time, ensuring you are never locked into a single hosting provider.

5. Company Dashboard

The Company Dashboard (`company_dashboard.html`) is the Command Center.

5.1 Managing Schools

This panel allows the creation of client accounts. Important actions:

  • Add School: Creates a new tenant partition in the database.
  • Lockdown Mode: The "Lock" icon toggles the is_locked flag. When active, the School Principal cannot modify any measurements. Use this after the "Final Measurement Date" has passed.

5.2 Activity Logs

The "Logs" tab is the "Black Box" of the system. It records:

  • Who: Username and IP.
  • What: Specific action (e.g., "Deleted Student ID 554").
  • When: Timestamp to the second.

7. Packing & Logistics Engine

The final step before dispatch. This module ensures 100% accuracy in deliveries.

7.1 The "Pack Visible" Algorithm

To speed up logistics, the "Pack Visible" button performs a bulk transaction:

1. It identifies all currently filtered students (e.g., "Class 10-A, House Red").
2. It verifies their Production Status is "Finished".
3. It updates their `order_status` to "Packed" in a single ACID transaction.
4. It generates a packing list PDF instantly.

7.2 Sticker Generation

The system generates distinct PDF stickers for every uniform bag. Each sticker contains: Student Name, Class, School Name, and a unique QR Code for tracking.

8. Planet Editor (Pattern Lab)

The Planet Editor is a browser-based spreadsheet designed for the Pattern Master.

8.1 Math Engine & Auto-Calc

The editor doesn't just store numbers; it understands them. When a pattern type (e.g., "Shirt Size 32") is selected, the system:

  • Looks up the "Cloth Consumption" factor (e.g., 1.25 meters).
  • Multiplies by the number of students.
  • Displays the Total Fabric Required in real-time.

9. Advanced Workflows

9.1 Disaster Recovery (Process)

If a school deletes their student list by mistake:

  1. Admin logs into Company Dashboard.
  2. Navigates to "Recycle Bin".
  3. Locates the deleted batch (retained for 5 days).
  4. Clicks "Restore".
  5. Data reappears instantly in the school's portal.
